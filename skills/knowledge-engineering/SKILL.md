# Knowledge Engineering Skill

将散乱的领域信息（文档、对话、代码、业务规则）提炼为结构化、可检索、可演进的知识资产。

> 工作流是管道，知识才是流过管道的活水。模型会迭代、工具链会更新、工作流会重构，但领域模型、架构决策、最佳实践、已知陷阱——这些是可累积的复利资产。

## 边界条件

- ✅ 适用：业务规则、技术方案、项目复盘、会议纪要、代码经验、流程规则、知识库治理
- ⚠️ 谨慎：时效性强、参数频繁变化的运营/策略知识
- ❌ 不适用：法律/医疗/财务审计/合同定稿/实时政策行情
- 存在冲突/不足时，输出"待澄清/待验证/冲突候选"，不强行结构化

## 知识三形态

| 形态 | 价值 | 示例 |
|------|------|------|
| **散点型** | 低 | "MySQL 默认隔离级别是 RR" |
| **因果型** | 中 | "广告预算高并发下超扣，因非原子操作 → 需 Redis+Lua" |
| **时空型** | 高 | "大非农前30分钟流动性骤降，不宜执行大额对冲" |

## 三维正交模型

每条知识必须在三个维度上明确定位：

### 维度一：存储层（知识住在哪里）

| Layer | 名称 | 范围 |
|-------|------|------|
| L0-P | 个人偏好 | 纯本地（编码风格、工具偏好） |
| L0-T | 团队约定 | 团队级（代码规范、Review 标准） |
| L1 | 技术知识 | 跨项目通用（Spring Boot 多租户模式） |
| L2 | 业务知识 | 按领域（B-Book 风控体系） |
| L3 | 项目知识 | 仅当前项目有意义 |

**提升规则：** L3 项目特有？留 L3。通用技术？提升 L1。通用业务？提升 L2。

### 维度二：知识类型（MECE，每条仅属一个）

| 类型 | 定义 | content 模板 |
|------|------|-------------|
| **model** | 实体/结构/关系 | 【实体/结构】【组成要素】【边界】 |
| **decision** | 技术选型/架构决策 | 【背景】【决策】【理由】【代价】 |
| **guideline** | 推荐(recommend)/禁止(avoid) | 【规则】【原因】【示例】【例外】 |
| **pitfall** | 已知风险/故障模式 | 【现象】【根因】【解法】【检测】 |
| **process** | 业务流程/状态机 | 【触发条件】【步骤】【异常分支】【终态】 |

**类型边界裁决：**
- decision vs guideline → 有明确取舍+理由 = decision；通用建议 = guideline
- pitfall vs guideline(avoid) → 有真实故障案例 = pitfall；预防性建议 = guideline
- model vs process → 静态结构 = model；有时序/步骤 = process
- **黄金测试：** "Agent 在什么决策点上会查询这条知识？"

### 维度三：成熟度

| 等级 | 条件 | 置信度 |
|------|------|--------|
| draft | 新提取，单一来源 | 0.5-0.6 |
| verified | 1 个项目/场景中验证 | 0.7-0.8 |
| proven | ≥2 个不同项目中验证 | 0.9+ |

**衰减机制：** proven 12月未引用→verified；verified 6月未引用→draft；draft 持续未引用→归档

## 知识条目标准格式

```yaml
id: "{PREFIX}-{DOMAIN}-{SEQ}"   # TK-SB-003, BK-AD-P001
title: "简明标题"
type: model | decision | guideline | pitfall | process
layer: L0-P | L0-T | L1 | L2 | L3
domain: "所属领域"
maturity: draft | verified | proven
polarity: recommend | avoid        # 仅 guideline
tags: ["标签1", "标签2"]
summary: "一句话摘要（≤30字）"
content: |
  # 按类型使用对应内部模板（见上）
evidence:
  contributors: ["贡献者"]
  source_references: ["来源"]
  last_referenced: "最后引用时间"
  projects_validated: ["验证项目"]
```

**命名规则：**
- L1 技术：`TK-{子领域}-{序号}`
- L2 业务：`BK-{领域}-{类型首字母}{序号}`
- L3 项目：`PK-{项目}-{序号}`

## 提炼工作流

### 阶段零：输入评估

| 输入规模 | 产出策略 | 条目上限 |
|---------|---------|---------|
| <2000字 | 直接条目 + 索引 | ≤5 |
| 2000-10000字 | 条目清单 + A/B索引 | ≤15 |
| >10000字 | 条目 + 三级索引 + 图谱 | ≤30（首批） |

超限时优先保留因果型和时空型，削减散点型。

### 阶段一：入库门槛过滤

```
信息点 →
  Q1: 可复用？→ 否 → 丢弃
  Q2: 已是常识？→ 是 → 丢弃
  Q3: 可验证/可执行？→ 否 → 弱候选
  Q4: 信息密度够？→ 否 → 拆分或降级为 evidence
  → 全部通过 → 入库候选
```

**反模式：** 纯观点、临时性事实、过度具体配置值、模型可零成本推导的常识 → 不入库

### 阶段二：结构化提取

对每条候选执行：
1. MECE 检查（与已有条目重复/矛盾？）
2. 层级判定（L1/L2/L3？）
3. 类型归属（严格五选一）
4. 边界标注（什么条件成立/不成立？）
5. 成熟度评估（draft/verified/proven）

### 阶段三：索引生成（三级渐进式）

| Layer | 内容 | 行数 |
|-------|------|------|
| **A 全景目录** | 统计 + 分类索引路径 | ~50行 |
| **B 分类清单** | ID/标题/类型/成熟度/标签 | ~100-300行 |
| **C 完整条目** | 按需阅读 | 每条50-200行 |

### 阶段四：Lint 质量审查

| 检查项 | 处理 |
|--------|------|
| 索引与条目不一致 | 自动修复 |
| 孤儿条目 | 标记 draft |
| 矛盾检测 | 标记冲突，供人裁决 |
| 重复/高度相似 | 标记合并候选 |
| 过时检测 | 标记需验证 |

## 知识图谱（条目≥10条时生成）

**建图策略：**
- 总节点 20-80 个（<20稀疏，>80混乱）
- Level 0 仅 1 个，Level 1 控制 3-6 个
- 边数 ≈ 节点数 × 1.2-1.8
- 每节点入度+出度 2-8
- 选择 5-8 种核心节点类型（不必全用11种）

## 输出规范

| 输出形式 | 适用场景 |
|---------|---------|
| **知识条目清单** | 默认，直接入库 |
| **知识图谱 JSON** | 领域全景理解 |
| **Markdown 知识文档** | 团队 Wiki / Git 仓库 |
| **永久记忆提炼** | 跨会话复用（每条≤500字符） |

## 硬约束（违反即返工）

1. 每条知识必须有**边界条件**
2. content 必须使用**对应类型的内部模板**
3. summary **≤30字**
4. 禁止伪知识入库
5. **先骨架后血肉** — 先输出 ID + summary + type 骨架，确认后再展开
6. **文件系统即状态机** — 所有知识都是 Markdown 文件，可见、可版本化
