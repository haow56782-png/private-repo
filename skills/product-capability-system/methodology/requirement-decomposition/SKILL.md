# 需求拆解方法

## 一、适用场景

- 模糊/一句话需求需要细化
- 大型需求需要拆分为可交付单元
- Sprint/迭代规划前的需求梳理
- 跨团队依赖识别
- 估算工作量前的需求分层

## 二、核心模型

### 2.1 需求拆解漏斗

```
一句话需求
    ↓
Epic（大功能）
    ↓
Feature（功能特性）
    ↓
User Story（用户故事）
    ↓
Task（开发任务）
    ↓
Acceptance Criteria（验收条件）
```

### 2.2 MoSCoW 优先级

| 等级 | 含义 | 策略 |
|---|---|---|
| **M**ust have | 必须有，否则产品不可用 | MVP 必须包含 |
| **S**hould have | 应该有，但不影响核心 | 优先级2 |
| **C**ould have | 可以有，锦上添花 | 看资源 |
| **W**on't have | 这次不做 | 明确排除 |

### 2.3 四层拆解法

```
产品逻辑：这个功能解决什么问题？（场景/价值）
交互逻辑：用户怎么操作？（流程/UI）
数据逻辑：数据从哪里来，到哪里去？（字段/状态）
工程逻辑：哪些依赖/约束/风险？（技术选型/性能）
```

## 三、设计步骤（SOP）

1. 接收原始需求 → 记录原话，不做加工
2. 追问 5W1H（Who/What/When/Where/Why/How）
3. 拆为 Epic → Feature → Story 三层
4. 为每个 Story 写验收条件（Given/When/Then）
5. 打优先级标签（MoSCoW）
6. 识别依赖关系（前驱/后继/阻塞）
7. 拆分到最小可交付单元（MVF）
8. 团队估算 + 排期

## 四、标准模板

### 用户故事格式

```
As a [角色]
I want to [功能]
So that [价值]

Acceptance Criteria:
- Given [初始状态]
- When [用户操作]
- Then [预期结果]
```

### 需求拆解表

```
| ID | Epic | Feature | Story | 优先级 | 估算 | 依赖 |
|----|------|---------|-------|--------|------|------|
```

## 五、可复用案例

- 交易平台"我要做充值功能" → 拆为：选择币种/生成地址/确认入账/通知用户/账本记入
- 客服系统"用户要投诉" → 拆为：投诉入口/分类/流转/回复/升级
- 用户一句话"我要个排行榜" → 拆为：数据源/排名规则/刷新周期/展示层/点击详情
