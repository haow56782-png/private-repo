# 产品评审框架

## 一、适用场景
- 需求评审（PRD Review）
- 交互/视觉设计评审
- 技术方案评审
- 上线前验收评审（Go/No-Go）
- 项目复盘/迭代后评估

## 二、核心模型

### 2.1 四阶段评审体系
```
| 评审类型 | 时机 | 参与方 | 产出 |
|---------|------|--------|------|
| 需求评审 | PRD完稿 | PM+研发+设计+测试 | 需求确认 |
| 设计评审 | 设计完稿 | PM+设计+研发 | 设计确认 |
| 技术评审 | 技术方案完成 | 研发+架构+PM | 方案确认 |
| 上线评审 | 发布前 | 所有相关方 | Go/No-Go |
```

### 2.2 决策记录模板
```
决策编号: DEC-001
日期: 
决策内容:
决策理由:
备选方案:
影响范围:
决策人:
```

### 2.3 评审检查清单
```
□ 需求背景清晰，问题定义明确
□ 产品/交互/数据/工程四层逻辑完整
□ 异常路径有处理方案
□ 验收标准可量化、可测试
□ 技术方案评估过可行性和成本
□ 变更管理跟进到位
```

## 三、评审SOP
1. 评审前：提前72h发资料，确保与会者已阅读
2. 评审会：聚焦核心逻辑和争议点，不读文档
3. 记录待办：明确 action owner 和 deadline
4. 评审后：24h内发出会议纪要和待办清单
5. 跟踪闭环：下一轮评审前检查上一轮待办完成情况

## 四、标准模板
### 评审会议纪要
```
会议主题:
日期:
参与人:
决策:
待办:
  - [ ] Task 1 (owner, deadline)
  - [ ] Task 2 (owner, deadline)
下次评审时间:
```

## 五、可复用案例
- 大版本上线评审：检查功能完整度、回归测试、灰度方案、回滚预案
- 紧急修复评审：影响面评估、快速测试、监控预案、用户通知
- 季度复盘：目标达成率、延期原因、质量数据、改进计划
