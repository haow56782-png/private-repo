# PRD 交付规范

## 一、适用场景
- 新功能/新产品 PRD 撰写
- 需求评审会前准备
- 产品文档标准化
- 跨团队对齐（技术/设计/测试/运营）
- 版本迭代记录

## 二、核心模型

### 2.1 PRD 标准结构
```
1. 背景与目标
   - 为什么做（业务价值/用户价值）
   - 成功标准（怎么衡量做完了）

2. 产品逻辑（场景/规则）
3. 交互逻辑（流程/页面）
4. 数据逻辑（字段/状态/接口）
5. 工程逻辑（依赖/约束/风险）

6. 边界与异常
   - 什么不做（明确排除）
   - 异常处理（失败/超时/降级）

7. 验收标准（QA 准入条件）
8. 排期与发布计划
```

### 2.2 需求质量门禁
```
             ┌──────────────────┐
             │  需求提出         │
             └────────┬─────────┘
                      ▼
          ╔══════════════════════╗
          ║  门禁1: 价值判断      ║ ← 这个功能真的该做吗？
          ╚══════════════════════╝
                      ▼
          ╔══════════════════════╗
          ║  门禁2: 需求完整度    ║ ← 产品/交互/数据/工程都写了吗？
          ╚══════════════════════╝
                      ▼
          ╔══════════════════════╗
          ║  门禁3: 技术可行性    ║ ← 研发评估能做得出来吗？
          ╚══════════════════════╝
                      ▼
          ╔══════════════════════╗
          ║  门禁4: 验收标准      ║ ← QA 能测吗？通过标准是什么？
          ╚══════════════════════╝
                      ▼
                  进入开发
```

### 2.3 变更管理流程
```
需求变更 → 评估影响（范围/时间/成本）
         ↓
         ┌── 小变更 → 记录+知会团队
         └── 大变更 → 重新评审+调整排期
```

## 三、PRD撰写SOP
1. 明确背景：写清楚问题定义和业务价值
2. 拆四层：产品逻辑/交互逻辑/数据逻辑/工程逻辑
3. 写异常：反向思考所有可能出错的地方
4. 定验收：研发和QA能据此写测试用例
5. 过评审：产品/技术/设计/测试四方对齐
6. 归档：发布后更新PRD状态

## 四、标准模板
### PRD 头信息
```
| 项目 | 内容 |
|------|------|
| 版本 | v1.0 |
| 状态 | 评审中/开发中/已发布 |
| 负责人 | |
| 关联需求 | |
```

## 五、可复用案例
- 提现功能PRD：用户/管理员/财务三个角色的完整流程
- 分销系统PRD：结算规则/佣金计算/提现限制
- 风控系统PRD：规则配置/事件处理/人工审核
